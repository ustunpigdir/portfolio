Featured case study

Controlled LLM Benchmark for Partial Quantum Search Reasoning

Clean release run in preparation

A local evaluation pipeline for testing whether language models can follow controlled quantum-search assumptions, preserve formulas, update downstream calculations, and expose interpretable failure modes.

Problem

LLM answers can appear mathematically fluent while violating a hidden assumption, changing a counting convention, dropping a correction term, or using the wrong baseline. The benchmark is designed to expose these specific failure modes.

Benchmark scenarios

PQS_BASE_001

Baseline partial quantum search calculation

PQS_VAR_SEQ_001

Sequential vs parallel local-search cost

PQS_CONSISTENCY_001

Inconsistent equal-block assumption

PQS_SCALE_001

Astronomical scale and correction-term preservation

PQS_COUNT_001

Changed final-inversion counting convention

Evaluation pipeline

Question → Model response → Deterministic numeric check → Sonnet-assisted autograding → taxonomy-v2 labels → Human review → Failure analysis export

Why it matters

The project does not try to prove which model is universally best. It identifies repeatable reasoning failure modes under controlled variations of the same mathematical structure.

Provenance

Research Source and Gold-Answer Verification

The benchmark scenarios are derived from a published discussion of partial quantum search and Grover-style query counting. The original research source was used to identify the mathematical structure of the problem: global Grover iterations, local block searches, final inversion conventions, and query savings relative to standard Grover search.

Primary research source

Placeholder

V. E. Korepin, “Partial Quantum Search of a Database,” arXiv:quant-ph/0504157.

arxiv.org/abs/quant-ph/0504157

Note

The benchmark questions are not copied from the paper. They are controlled scenario variants created from the paper’s mathematical setup to test specific LLM failure modes: parallel-vs-sequential local search, changed counting conventions, inconsistent block assumptions, scientific-notation scaling, and correction-term preservation.

Gold-answer verification

Gold answers were computed and checked independently using Python-based scientific calculation tools, including symbolic and numerical libraries such as SymPy, NumPy, SciPy, and mpmath where appropriate. The core numerical answers were also manually verified step by step before being used as answer keys.

Research source → Scenario design → Python calculation → Manual verification → Gold answer → Model evaluation

Transparency

This workflow separates source citation, scenario construction, numeric answer-key generation, and final human annotation. The goal is to make the benchmark reproducible and auditable rather than relying on unverified model-generated answer keys.

Reproducibility note

  • Original mathematical setup is cited.
  • Scenario variants are documented separately.
  • Gold numeric values are stored as expected-value fields.
  • Tolerance mode is specified for each question.
  • Python verification is used for calculation checks.
  • Final labels are human-reviewed.

Design principles

  • Closed-book prompts for the clean English run
  • Same system prompt across models
  • Fixed temperature and trial count
  • Autograding is advisory, human review is final
  • Pipeline artifacts are separated from model failures

Approach

Methodology

The pipeline is designed around interpretability of failures, not just aggregate scores.

Controlled prompt variants

Each scenario shares the same mathematical structure but changes one assumption, baseline, or convention. This isolates which variation a model actually mishandles instead of blending many effects together.

Gold numeric answer keys

Every scenario has a precomputed reference value with an explicit tolerance. Numeric checks are deterministic, so a model's arithmetic can be judged independently of its wording.

Required-concept checklist

A short checklist records which assumptions, formulas, and correction terms a correct answer must contain. This separates conceptually valid reasoning from fluent text that skips a step.

Grouped error taxonomy

Failures are tagged with a structured, grouped taxonomy rather than a flat label list, so conceptual, numeric, response-quality, and pipeline issues stay distinct and countable.

Autograding + human review

An autograder proposes labels and rationale to speed up triage, but a human reviewer makes the final call. Disagreements between automated and human labels are tracked.

Failure-analysis exports

Results are exported as structured tables and representative cases. The goal is reusable evidence: failures that can be inspected, counted, and reproduced by others.

Source-grounded scenario design

The benchmark starts from a cited research source rather than arbitrary synthetic prompts. I use the source to identify the mathematical structure, then create controlled scenario variants that stress-test specific reasoning behaviors. Each variant is documented with its own gold answer, expected numeric values, tolerance settings, and required-concept checklist.

A wrong answer is only useful if the failure mode is specific enough to inspect, count, and reproduce.

Labeling system

Taxonomy v2: separating reasoning failures from numeric errors and pipeline artifacts

The project moved from a flat error-label list to a grouped taxonomy after pilot runs revealed that formula mistakes, rounding errors, incomplete responses, and extractor artifacts should not be treated as the same type of failure.

Conceptual labels

C_
  • C_PARALLEL_AS_SEQUENTIAL
  • C_WRONG_GROVER_BASELINE
  • C_CORRECTION_TERM_DROPPED
  • C_FALSE_SAVING_CLAIM

Core reasoning or assumption failures that usually determine whether the answer is conceptually invalid.

Numeric labels

N_
  • N_TOTAL_VALUE_ERROR
  • N_SAVED_VALUE_ERROR
  • N_PRECISION_INSUFFICIENT
  • N_SCIENTIFIC_NOTATION_ERROR

Calculation, tolerance, rounding, or scientific-notation errors.

Response-quality labels

R_
  • R_INCOMPLETE_RESPONSE
  • R_CONTRADICTORY_RESPONSE
  • R_EMPTY_RESPONSE
  • R_REPETITIVE_LOOP

Problems in answer form, completeness, consistency, or responsiveness.

Artifact labels

A_
  • A_NUMERIC_EXTRACTION_ARTIFACT
  • A_DUPLICATE_EXPORT_ENTRY
  • A_RUN_METADATA_MISSING

Pipeline or review artifacts that should be separated from model-performance statistics.

Example

If a model writes 10^80 but the numeric extractor reads 10, the case receives A_NUMERIC_EXTRACTION_ARTIFACT rather than being counted as a model arithmetic error.
